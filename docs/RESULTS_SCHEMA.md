# Results Schema — WFCRC Research Program (Paper 1)

> **Status:** Frozen for first use (MS8), **not yet reviewed by a domain
> expert / PI**. **Version:** 1.0. **Date:** 2026-07-18. **Scope:** every
> experimental output E1–E12 produces, from a single run's `manifest.json`
> up to the publication-ready figure/table numbering cited in
> `docs/EXPERIMENT_PROTOCOL.md` §3. This schema documents the **already-
> implemented** `wfcrc.runner`/`wfcrc.visualization` output formats
> (`Manifest`, `FigureFile`, `Checkpointer`) — it defines no new file
> format, only the stable convention for organizing, naming, and
> aggregating many runs' worth of those already-frozen artifacts across a
> full E1–E12 campaign. This schema must remain stable across all
> experiments (per the MS8 brief) — any change here is a versioned
> revision (§7), never a silent renumbering.

---

## 1 · Directory Layout

### 1.1 Per-run directory (already-frozen `ExperimentRunner` output)

```
<run_dir>/
  manifest.json              # Manifest.to_dict() — see §3
  checkpoints/               # Checkpointer's content-addressed stage cache
    <stage_key>.json         #   (calibration result, metrics, etc.; keyed by
                              #    stage_key(stage, *inputs) — a hash, not a
                              #    human-readable name)
  figures/
    <figure_name>.<pdf|svg>  # FigureFile.path — e.g. g_curve.pdf
    <figure_name>.csv        # FigureFile's data sidecar (source_hash header)
```

This is the unit `ExperimentRunner.run(...) -> ResultBundle` already
produces; MS8 adds no field to it. `run_dir` naming (below) is the one
convention this schema fixes that `ExperimentRunner` itself leaves to the
caller.

### 1.2 Campaign-level directory (MS8 convention, new)

```
results/
  E1/
    <run_id>/                # one <run_dir> per (dataset, family, alpha,
      manifest.json          #   seed) cell — see §1.3 for <run_id>
      checkpoints/
      figures/
    aggregated/
      T1.csv                 # aggregated table, see §6 numbering
      F1_<dataset>_<family>.pdf
      F1_<dataset>_<family>.csv
  E2/
    <run_id>/...
    aggregated/
      T2.csv
      F2_<dataset>.pdf / .csv
  ...
  E12/
    <run_id>/...
    aggregated/
      failure_catalog.jsonl  # E12 has no numeric table (§6); see §5
  campaign_manifest.json     # one record per <run_id> across the whole
                              #   campaign — see §4
```

- **`results/` lives outside `wfcrc/` and `tests/`** — it is generated
  output, not source, and is `.gitignore`d exactly like the existing
  top-level `data/` cache convention (`PROJECT_CONTEXT.md` §6). Only
  `campaign_manifest.json` and the `aggregated/` tables/figures are
  candidates for committing to a durable, citable artifact store
  (`docs/DATASET_SPLIT_POLICY.md` §6's "version-controlled... or an
  equivalently durable, citable artifact store" standard, applied here to
  results rather than splits) — per-run `checkpoints/` caches are
  reproducibility scaffolding, not a citable artifact.
- **One `results/E<n>/` directory per experiment**, never a shared pool —
  this mirrors the Experiment Matrix's own per-experiment structure
  (`docs/EXPERIMENT_PROTOCOL.md` §3) and keeps every experiment's raw runs
  independently re-aggregatable without touching another experiment's data.

### 1.3 `<run_id>` naming convention

```
<dataset>__<family>__a<alpha>__pi<pi>__seed<seed>__r<resample_index>
```

Example: `cityscapes__cvar__a0.10__pi0.30__seed42__r000`. Every field in
`<run_id>` is also present, machine-readably, inside that run's own
`manifest.json` (`config_hash` covers all of them) — the directory name is
a human-readable index into the manifest, never the sole record of a run's
identity. `r000`–`r099` spans the `R=100` resamples
(`docs/EXPERIMENT_PROTOCOL.md` §3 note); a single-run experiment cell
(e.g. E12's illustrative cases) omits the `__r<...>` suffix.

## 2 · CSV Schema

### 2.1 Figure data sidecars (already-frozen `FigureFile` contract)

Every rendered figure (`.pdf`/`.svg`) ships a `.csv` sidecar that is
**exactly** the data used to render it (`wfcrc.visualization.base` module
docstring, MS5 spec C1 items 4–9) — this is not a new schema, it documents
the existing one:

- Header row: series/column names as used by the corresponding `_build_*`
  function in `wfcrc.visualization.plots` (e.g. for F1: `alpha, family,
  realized_risk, ci_lo, ci_hi`).
- A `source_hash` provenance line (per the module docstring) ties the
  sidecar back to the exact metrics manifest it was rendered from — a
  figure can be regenerated byte-for-byte from that manifest with no
  algorithm re-run.
- One sidecar per figure file, same basename, `.csv` extension.

### 2.2 Aggregated table CSVs (`results/E<n>/aggregated/T<n>.csv`, MS8 convention)

Column convention, common to T1–T7 (dataset/family/method columns vary per
table per `docs/EXPERIMENT_PROTOCOL.md` §3; the statistic columns are
fixed):

| Column | Meaning |
|---|---|
| `experiment` | `E1`–`E12` |
| `table` | `T1`–`T7` (§6 numbering) |
| `dataset` | Phase-A dataset key, matching `wfcrc.datasets.metadata.DATASET_METADATA`'s own keys (`cityscapes`, `cityscapes_c`, `acdc`, `msd_hippocampus`, `msd_pancreas`, `cifar10`, `cifar10_1`, `kvasir_seg`) — never a free-text dataset name |
| `family` | `cvar` / `kl` / `finite_group` / `known_weight` / a named baseline (`vanilla_crc`, `gibbs`, `aa_crc`, `sem_crc`, `pooled_kfold`, `total_n_inflation`, `fixed_eta`, ...) |
| `alpha` | Target risk level (blank/`NaN` for non-α-indexed rows, e.g. E10 timing) |
| `metric` | Metric name, matching `docs/EXPERIMENT_PROTOCOL.md` §6's names exactly (`realized_worst_case_risk`, `realized_marginal_risk`, `per_group_max_risk`, `mean_set_size`, `coverage`, `duality_gap`, `n_eff`, ...) — never a paraphrase |
| `mean` | Aggregated mean across `R` resamples |
| `ci_lo`, `ci_hi`, `ci_level` | Bootstrap CI bounds and nominal level (§5.1 of `EXPERIMENT_PROTOCOL.md`) |
| `n_resamples` | `R` actually aggregated (should equal 100 unless a documented `SweepCellFailure`, MS5, reduced it) |
| `stat_test`, `stat_statistic`, `stat_p_value`, `stat_p_holm` | Populated only where §5.2/§5.3 of `EXPERIMENT_PROTOCOL.md` apply to that row; blank otherwise |
| `config_hash_set` | The set of `config_hash` values (semicolon-joined) this aggregated row summarizes — traces every aggregated number back to its exact constituent runs |

No table row may report a `mean` without its paired `ci_lo`/`ci_hi` (per
`EXPERIMENT_PROTOCOL.md` §5.5's "never a point estimate alone").

## 3 · JSON Schema

### 3.1 Per-run `manifest.json` (already-frozen `Manifest.to_dict()`)

```json
{
  "config_hash": "string",
  "seed": 0,
  "family_type": "cvar | kl | finite_group | known_weight",
  "family_params": {"...": "..."},
  "git_commit": "string | null",
  "environment": {"...": "..."},
  "n_a": "int | null",
  "n_b": "int | null",
  "b_tilde": "float | null",
  "r_hat_b": "float | null",
  "lambda_hat": "float",
  "empty_flag": "bool",
  "diagnostics": {"...": "..."},
  "verification_passed": "bool | null",
  "metrics": {"...": "..."},
  "figure_paths": {"figure_name": "relative/path"}
}
```

This is `wfcrc.runner.runner.Manifest`'s own schema, reproduced here
verbatim for the methodology record (§0 note) — no field is added or
renamed by MS8. `verification_passed` is always `true` for any manifest
that exists on disk (a failing STOP-gate halts `run()` before writing one,
per `docs/EXPERIMENT_PROTOCOL.md` §2.1) — its presence is a self-describing
record, not a filter a downstream aggregator must apply.

### 3.2 Campaign manifest (`results/campaign_manifest.json`, MS8 convention)

```json
{
  "campaign_id": "string",
  "campaign_base_seed": "int",
  "created": "ISO-8601 timestamp",
  "protocol_version": "docs/EXPERIMENT_PROTOCOL.md version, e.g. 1.0",
  "runs": [
    {
      "run_id": "string, per §1.3",
      "experiment": "E1..E12",
      "manifest_path": "results/E<n>/<run_id>/manifest.json",
      "config_hash": "string, redundant with the manifest's own field, indexed here for fast lookup"
    }
  ]
}
```

One `campaign_manifest.json` per full E1–E12 execution attempt; a
re-execution (e.g. after resolving one of `EXPERIMENT_PROTOCOL.md` §8's
open items) produces a **new** `campaign_id`, never an in-place overwrite —
same discipline as `docs/DATASET_SPLIT_POLICY.md` §6's `SplitManifest`
versioning.

## 4 · Figure Naming

Figure basenames follow `F<n>` from the Experiment Blueprint (§26),
qualified by dataset/family where a figure is produced per-cell rather than
once per campaign:

| Figure | Name (Blueprint §26) | Filename convention | Produced by |
|---|---|---|---|
| F1 | Realized worst-case risk vs `α` (all families) | `F1_<dataset>_<family>.pdf`/`.csv` | `wfcrc.visualization.plots.plot_risk_vs_alpha` |
| F2 | Per-region realized risk, WF-CRC vs vanilla (bars) | `F2_<dataset>.pdf`/`.csv` | `plot_group_risk` |
| F3 | Realized risk vs corruption severity (WF-CRC vs vanilla vs robust-CP) | `F3_<dataset>_synthetic.pdf` (E3) / `F3_<dataset>_real.pdf` (E4) | `plot_risk_vs_shift` |
| F4 | Set-size vs `α` efficiency frontier | `F4_<dataset>.pdf`/`.csv` | `plot_set_size` |
| F5 | Coincide/diverge nesting | `F5_<dataset>.pdf`/`.csv` | `plot_nesting` |
| F6 | Qualitative segmentation heatmaps | `F6_<dataset>_<case_id>.pdf`/`.csv` | `plot_qualitative` |
| F7 | Runtime/memory scaling | `F7_<dataset>.pdf`/`.csv` | `plot_runtime` |
| F8 | Reliability curves | `F8_<dataset>.pdf`/`.csv` | `plot_reliability` |
| — | Single-run `g`-curve (dual families only, diagnostic, not a numbered paper figure) | `g_curve.pdf`/`.csv`, inside the per-run `figures/` directory only (§1.1) | `plot_g_curve`, called automatically by `ExperimentRunner.run` |

`<dataset>`/`<family>`/`<case_id>` use the same canonical keys as §2.2's
table columns — a figure and a table describing the same result must use
identical key strings, never a differently-cased or abbreviated variant
(this is what §8's cross-reference quality gate checks).

| Table | Content | Owning experiment(s) |

| T5 | Hyperparameter sensitivity | E9 |
| T6 | Runtime/memory | E10 |
| T7 | Full baseline-suite comparison | E6 (matched-validity set size), general baseline suite |

JSON line per documented failure case
(`{case_id, dataset, failure_mode, description, figure_ref}`), never
coerced into a numeric table it was never meant to be.

## 7 · Aggregation Rules

- **Aggregation unit:** one `(experiment, dataset, family, α, π)` cell,
  aggregated across its `R` resample seeds (`docs/EXPERIMENT_PROTOCOL.md`
  §3 note) — never across datasets, families, or α values, which are kept
  as separate rows/series per §2.2/§4.
- **Aggregation statistic:** mean ± bootstrap 95% CI (`bootstrap_ci`,
  `n_resamples=2000`), per `docs/EXPERIMENT_PROTOCOL.md` §5.1 — no other
  aggregation statistic (median, trimmed mean, etc.) is used unless a
  future dated protocol amendment adds one.
- **Per-group aggregation:** `per_group_risk`'s per-group means are
  aggregated across resamples exactly like any other metric; the
  **max-over-group** step (the reported "conditional validity" number) is
  taken **after** per-resample per-group aggregation, not before — i.e.
  report `mean_over_resamples(max_over_groups(risk))`, not
  `max_over_groups(mean_over_resamples(risk))`, since the latter would
  silently understate worst-case-group risk by letting different resamples
  contribute their worst group inconsistently. This ordering must be
  applied identically by every aggregation script; a script computing it
  the other way produces a number that looks plausible but answers a
  different question (the same "silent, plausible-but-wrong" failure mode
  `MS6_ARCHITECTURE_SPEC.md` §7 R-GM1 already warns about for group masks).
- **Missing/failed cells:** a `SweepCellFailure` (already-frozen
  `wfcrc.runner.runner` type) reduces `n_resamples` for that cell below
  100; the aggregated CSV's `n_resamples` column must reflect the true
  count, and any cell falling below a to-be-determined minimum usable
  count is flagged, not silently averaged over a smaller-than-intended
  sample without disclosure (open item, §8 of `EXPERIMENT_PROTOCOL.md`
  implicitly extends here: no minimum `n_resamples` floor is yet frozen).

## 8 · Comparison Conventions

- **WF-CRC vs. baseline, within one experiment:** paired Wilcoxon
  signed-rank + Holm correction across baselines
  (`docs/EXPERIMENT_PROTOCOL.md` §5.2–§5.3), reported in the same
  aggregated table row set as the metrics being compared (`stat_*` columns,
  §2.2) — never in a separate, disconnected "statistics" file that could
  drift out of sync with the metric table it describes.
- **Matched-validity comparisons (E6):** set size is compared **only**
  across methods achieving realized risk within the same CI band of `α`
  (`docs/EXPERIMENT_PROTOCOL.md`'s E6 row: "compared at matched realized
  validity, not a fixed α") — an aggregated T1/T7 row for E6 must record
  which methods were excluded from a given comparison for failing to match
  validity, not silently drop them.
- **Ablation comparisons (E7):** one-sided `≤ α` test applied
  **independently per variant** (single-split / pooled-K-fold / total-n /
  fixed-η) — T4 reports all four variants' outcomes side by side, including
  the variants **expected to fail** (K-fold, total-n under-covering) so the
  negative result is itself part of the published record, not filtered out
  as an "error."
- **Cross-experiment comparisons are out of scope for this schema.** Each
  experiment's own comparisons stay within that experiment's own table
  (§6); nothing in this program compares, e.g., E2's per-group risk against
  E3's shift risk as if they were commensurable — doing so would conflate
  two different scientific questions.

## 9 · Publication-Ready Directory Organization

For a manuscript submission package, `results/` is distilled into a
smaller, camera-ready structure — never regenerated by hand, always
produced by re-running the aggregation step (§7) against the frozen
`results/E<n>/<run_id>/` raw runs, so the submission package is always
traceable back to `campaign_manifest.json`:

```
paper_artifacts/
  tables/
    T1.csv  T2.csv  ...  T7.csv
  figures/
    F1_cityscapes_cvar.pdf  F1_cityscapes_cvar.csv
    ...
  failure_catalog.jsonl
  campaign_manifest.json
  environment/
    lock.txt               # requirements/lock.txt, copied verbatim
    reproduce_golden.json  # tests/fixtures/reproduce_golden.json, copied verbatim
```

`paper_artifacts/` is the one directory this program's reproducibility
checklist (Experiment Blueprint §29) treats as the "released code +
configs + trained-model score caches" deliverable; it is built, not
hand-edited, so it can never silently diverge from the `results/` tree
that produced it.
